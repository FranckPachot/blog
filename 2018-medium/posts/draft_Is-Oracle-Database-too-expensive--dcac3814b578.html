Is Oracle Database too expensive?

This post, unusually for my blog, is more about thinking than technical facts. I’ve read this Hacker News thread about the Oracle Database source code, and discussed with friends in the community. 

Oracle Database 12.2. It is close to 25 million lines of C code. What an unima... | Hacker News
Very complex pieces of logic, memory management, context switching, etc. are all held together with thousands of flags…news.ycombinator.com

This gives some lights to the recurrent discussions about the price of Oracle Database, and the quality of support. I’m not always an advocate for Oracle pricing and support but after reading this feedback for a former developer on the database, I’ve some positive points to share… Quotes from the post are in italics.

>> Very complex pieces of logic, memory management, context switching, etc. are all held together with thousands of flags.

That is not a surprise. Large parts of the kernel have been written decades ago, working on a huge number of platforms (processors, OS, storage, network protocols,…). The first Oracle Database I installed was in version 7.1 on Sun SPARC workstations plugged to a TokenRing network. And most of this kernel code was probably already there. And they went to other platforms, adapt to different hardware, changed some architectural design, cope with very large databases… while keeping complete compatibility.

There are databases that have been upgraded since those versions, and old applications still running. And the magic is that you can still run the most recent versions with compatible behavior with those very old versions. the compatible parameter controls the format of the files so that what you store with recent version may be downgraded to older versions. optimizer_features_enable parameter controls the version of the optimizer so that you can run a query, a session or a whole application with same execution plan as a previous version. This is priceless when you upgrade because you can decorrelate the upgrade (which you do to get supported, have security issues fixed, and to benefit from new features) from the risk of performance regression.

Those many flags in the code are required for this, in order to map any change (bug fix or new feature) with a version.

>> After 2 weeks to 2 months, when everything is complete, the code would be finally merged into the main branch.

>> The above is a non-exaggerated description of the life of a programmer in Oracle fixing a bug

The description of the time it takes to understand the code, fix a bug, and ensure non-regression is very important to understand the price of Oracle Support. support is not only the interaction with the support engineers when you open a ticket.
